# Scenario Validation

本文定义 scenario 准入规则。目标是区分开发用 smoke scenario 和可进入正式 benchmark 的 official scenario，避免任务契约不完整导致评测结果不可解释。

## Scenario Levels

### Smoke Scenario

用于开发、调试和端到端连通性测试。

允许：

- observation/action meanings 不完整；
- episode 数较少；
- reward normalization 未定；
- baseline policy 很弱。

限制：

- 不能进入 leaderboard；
- 不能作为论文或正式 benchmark 结果；
- report 必须标记 `scenario_level="smoke"`。

### Official Scenario

用于正式评测、横向比较和报告。

必须满足本文件所有 official requirements。缺任何 P0 条件都应拒绝注册为 official。

## Required Files

每个 scenario 目录至少包含：

```text
src/hlbench/scenarios/<scenario_name>/
  scenario.json
  policy.py
  task_spec.md
  README.md
```

要求：

- `scenario.json` 是机器可读配置。
- `policy.py` 是 baseline policy，必须可编译、可导入、可执行。
- `task_spec.md` 是人类可读任务补充说明。
- `README.md` 包含 smoke command 和 scenario 维护说明。

## Required Scenario Fields

Official `scenario.json` 必须包含：

```json
{
  "name": "cartpole_balance",
  "scenario_id": "cartpole_balance_v0",
  "scenario_level": "official",
  "env_backend": "gymnasium",
  "env_id": "CartPole-v1",
  "env_kwargs": {},
  "observation_mode": "jsonable",
  "observation_options": {},
  "task": {
    "goal": "Keep the pole balanced.",
    "success_condition": "Episode reaches max_steps.",
    "reward_description": "Environment reward per step while balanced."
  },
  "observation_meanings": [],
  "action_meanings": [],
  "max_steps": 500,
  "splits": {
    "train": {"seed_pool": "default/train", "public_feedback": true},
    "validation": {"seed_pool": "default/validation", "public_feedback": false},
    "heldout": {"seed_pool": "default/heldout", "public_feedback": false}
  },
  "minimum_score": {
    "performance_score": 0.0,
    "total_reward": -1.0
  }
}
```

## Observation And Action Semantics

Official scenarios must provide enough semantic information for an agent to understand the public interface.

Rules:

- `EnvContract` may infer type, shape, dtype, low/high from environment spaces.
- `ScenarioSpec` must provide meanings for fields or dimensions when the backend cannot infer them.
- `observation_mode` must match the declared public information surface. Gymnasium Classic Control scenarios are telemetry by default and should use the official numeric observation unless registered as a separate pixel-control variant.
- Numeric `jsonable` observations are acceptable when the scenario is explicitly framed as public telemetry, so humans and policies reason over the same sensor readings.
- If a task needs both visual and telemetry variants, register separate scenarios rather than mixing surfaces in one task.
- `action_meanings` are required for discrete actions.
- continuous action dimensions must include names and meanings.
- `Dict` and `Tuple` spaces must describe each field/item recursively.

Automatically inferred shape alone is insufficient for official status.

## Split Requirements

Official scenarios must reference deterministic shared seed pools for train / validation / heldout splits.

Default minimums:

```text
train:       >= 1000 seeds for official scenarios
validation: >= 100 seeds
heldout:    >= 100 seeds
```

Exceptions are allowed only for expensive environments and must be documented in `scenario.json`.

Rules:

- Shared seed pool files must exist and load as JSON objects with a `seeds` list.
- Seed files should be generated by a fixed `generator_seed` using random partitioning, not predictable contiguous ranges.
- Splits must not overlap.
- `--episodes` samples from a split seed pool and does not define the split.
- `generator_seed`, seed pool names, seed file paths, and seed values must not be rendered into `task_contract.json`.
- Heldout seeds must never appear in workspace, task contract, report, prompt, or public feedback.
- Validation seeds must not appear in workspace, task contract, prompt, or public feedback.
- Seed order must be stable.

## Reward And Success

Official scenarios must define:

- primary score source, usually environment return;
- success condition;
- timeout/truncation semantics;
- minimum score;
- whether reward is raw or normalized.

Single-environment reports must include raw return. Cross-environment leaderboards should use normalized metrics, but normalization must be scenario-defined before publishing aggregate rankings.

## Baseline Policy Validation

Baseline `policy.py` must:

- compile without syntax errors;
- expose the required `Policy` interface;
- return actions valid under `ActionSpec`;
- run at least one train rollout without execution error;
- not access private data or external evaluator state.

A weak baseline is acceptable. A broken baseline is not acceptable for official scenarios.

## Visibility Validation

The validator must reject official scenarios if:

- `task_contract.json` would include validation / heldout seeds;
- validation / heldout evaluator would generate replay or per-episode artifacts;
- workspace feedback contains non-train data;
- report artifacts are copied into checkpoint workspace;
- private evaluator config is agent-readable.

## Validation Checklist

Before marking a scenario official:

- `scenario.json` validates against `scenario.schema.json`.
- `EnvironmentBackend.describe()` produces a complete `EnvContract`.
- `TaskContractBuilder` writes `task_contract.json` and `task.md`.
- observation/action meanings meet official requirements.
- baseline policy compiles and imports.
- train smoke rollout succeeds.
- validation / heldout evaluation produces aggregate summaries only.
- no validation / heldout replay or per-episode files are generated.
- visibility validator passes.
- minimum score behavior is configured.

## Rejection Reasons

Common rejection reasons:

- missing task goal, success condition, or reward description;
- missing action meanings;
- missing observation field/dimension meanings;
- overlapping seed splits;
- missing or broken baseline policy;
- private data appears in task contract or workspace;
- validation / heldout evaluator generates replay;
- reward scale or success semantics are undefined.
