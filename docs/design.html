1. 这是评测什么 #

给定一个控制任务、一个紧的 rollout 预算、一个跨 iteration 持久的 workspace, 编码 Agent 能多好地改进它提交的策略代码?

评测分数来自一个隐藏的 held-out episode 池 —— 用一个独立、agent 不可见的 seed pool 跑最终策略并打分。

关键测量目标

  • iterative code refinement —— Agent 不是一次性产物,而是 read-feedback / edit-code 多轮循环
  • tight rollout budget —— 默认 256 episodes 总预算,逼出「每一次 rollout 都贵」的策略选择
  • method neutrality —— heuristic / classical control / 神经网络 / search 都允许;不规定如何解
  • generalization —— 最终用 disjoint hidden seeds 评测,排除 in-loop 过拟合

3. 核心不变量 #

Core Invariant. Agent 与其提交的策略所使用的所有环境信息,必须流经 submit-feedback 循环;最终提交的策略必须在 disjoint、hidden seed pool 上泛化。

这一条是设计的中枢。后面所有规则(/info 不暴露 baselines、env 用 ID 寻址、held-out 完全不可见、denied_imports 强制等)都是把这条 invariant 落地的 机制。

4. 端到端测试流 #

① OPERATOR

起 harness

hlbench init + serve,workspace 与 HTTP server 就绪

② AGENT

探查

GET /info、GET /task、读 AGENTS.md

③ AGENT

首次提交

写 system/policy.py,POST /submit

④ AGENT

迭代

读 feedback,改代码,再 submit,直到预算耗尽

⑤ AGENT

finalize

POST /finalize 触发隐藏 held-out 评测

⑥ OPERATOR

取分

读 runs/<…>/run.json:final_score

每一步都同步阻塞:POST /submit 阻塞到 N episodes 全部跑完 + feedback 落盘 才返回。POST /finalize 阻塞到 held-out 评测全部跑完 + run.json 落盘才返回。 没有长轮询、没有 tar bundle —— 共享文件系统 + 同机 HTTP 是协议的永久承诺 (CLAUDE.md invariant 8)。

6. Policy 接口 #

system/policy.py 必须定义一个名为 Policy 的类, 实现以下 4 个方法 (SPEC §2):

class Policy:
    def __init__(
        self,
        obs_space: gymnasium.Space,
        action_space: gymnasium.Space,
        env_meta: Mapping[str, Any],
    ) -> None:
        """每个 submit 构造一次,跨 episode 共享。
        env_meta 含 submit_index / n_episodes_this_submit /
        remaining_budget_after / max_episode_steps 等。"""

    def reset(self, episode_index: int) -> None:
        """每个 episode 开始时调用。episode_index 是 submit 内的本地索引
        (0..n_episodes_this_submit-1)。env 真实 seed 不暴露,不可推断。"""

    def act(self, obs) -> action:
        """每步调用一次。返回 action_space 合法的 action。
        默认 wall-time 上限 10ms (env 可放宽)。"""

    def on_episode_end(self, episode_return: float) -> None:
        """可选。episode 结束时调用,可用来更新跨 episode 持久状态
        (写 system/ 文件)。"""

状态持久化的两个层级

  • Submit 内: Policy 实例的 attribute 跨 episode 持久 (init 一次,reset/act/on_episode_end 多次)
  • 跨 submit: Policy 实例销毁。要持久必须写 system/ 下的文件,下次 submit init 再读回

7 阶段 (docs/submit-protocol.md §2)

8. Feedback 工件 #

feedback/submit_NNN/
├── summary.json                 # submit 级聚合 (SPEC §4.1)
├── errors.txt                   # 仅 submit 级失败时;JSONL,64KB cap
└── episodes/                    # 与 errors.txt 互斥(oom/wall 例外)

        ├── stdout.txt           # 0.1.0a1: 始终创建,64KB cap
        ├── stderr.txt           # 同上
        ├── observations.npy     # 仅 obs_storage=external
        ├── video.mp4            # 仅 env 可 render
        └── error.txt            # 仅本 episode 中途失败

summary.json 关键字段

字段类型说明
statusstring11 verdict 之一
n_episodesint本次请求数(=扣预算数,不论成败)
first_global_episodeint|nullreturns[0] 对应的 run-global 索引
env_instancesint[]本次请求的 env_instance ID 列表(顺序一致)
remaining_budgetint扣完后的剩余
returns / mean_return / std_return / min / maxfloat[] / float本 submit 内 raw episode return

reward_components_per_episodeobject|null仅 env 声明 reward_components 时

所有「失败时为 null」的字段在 status != "ok" 时用 null 而非空数组。这样 status == "ok" 是「这次 submit 跑了吗」的 trivial 判定。

Episode 索引 (SPEC §4.0)

Episode 索引是 run-global 而非 submit-local —— 一次 run 内全程递增。这让「ep_142」这种引用全局唯一,replay/backtest/cross-submit join 都不会歧义。每尝试一个 episode 计数器加 1,失败的 submit (没跑任何 episode) 不消费索引。

Trajectory schema (SPEC §4.2)

{"t": 0,   "obs": [...], "action": [...], "reward": 0.12,
 "terminated": false, "truncated": false, "info": {}}
{"t": 1,   "obs": [...], "action": [...], "reward": 0.15, ...}
...
{"t": 998, ...}

Field 顺序固定方便 grep。NaN/Inf 编码为 "NaN" / "Inf" / "-Inf" 字符串。action 记的是 policy 返回值(pre-clip),env 的 clipping 是确定性变换,可由 action_space 复原。

9. 打分公式 #

Final Score (Headline) — SPEC §5.2

final_submit = .final_submit override or 最近一个 status=="ok" 的 submit
mean_held_out = mean over M held-out episodes (M default 256, hidden)

normalized = (mean_held_out - random_baseline)
           / (expert_baseline - random_baseline)
final_score = clip(normalized, 0.0, 1.2) * 100
  • random_baseline: uniform-random policy 在同 M 个 held-out seed 上的平均
  • expert_baseline: env 注册时声明的 expert 水平
  • 两个 baseline 都是 server-internal:agent 在任何时候(/info、/task、feedback、run.json)都看不到。这迫使 agent 在不知道目标阈值的情况下优化
  • 上界 clip 到 1.2 (=120 分):允许超 expert 表现登记,但限制 outlier 拉爆均值

Auxiliary Metrics — SPEC §5.3

指标定义用途
auc_in_loop累计 episode × normalized_score 的梯形 AUC, (0,0) 锚定,/总 episode 数,缩放到 [0,100]同预算下 agent 比较;不适合跨预算比较(锚点结构性偏置小预算)
episodes_to_50pct累计 episode 数到首次 normalized_score ≥ 0.5「学到一半」
episodes_to_80pct同上,≥ 0.8「学到 80%」
held_out_gapmean_in_loop_final − mean_held_out_final正 = in-loop 好但 held-out 差(过拟合可能);负 = 反之
n_submits / n_successful_submits提交次数(含失败 / 仅成功)—
episodes_used / mean_episodes_per_submit消耗预算 / 平均每次 submit 大小策略颗粒度
Day 14 spec 修正 早期 SPEC 把 episodes_to_Npct 写成 mean_return ≥ 0.5 × expert, 在负奖励 env (Pendulum expert ≈ −150) 下变成「比 expert 还好」的不可达阈值。 现规约为 normalized form:normalized_score(mean) ≥ N。

10. Held-out 隐藏机制 #

两个 disjoint 静态 seed 池

文件用途Agent 可见?
envs/<id>/train.jsonN 个 agent 可寻址 env_instance 的 real seed仅个数 (n_env_instances) 可见;ID→seed 映射不暴露
envs/<id>/heldout.jsonM 个 held-out 评测 seed完全不可见(包括池大小、seed 值、单 episode return)

两个池由 env 作者用单一 master_seed 一次性生成、commit 进 env 包、与 env_version 绑死。改 seed 池 = bump env_version。

Seed 间接寻址

agent submit --env-instances 5
                    │
                    ▼
   harness 读 train.json[5] → real_seed = (例如) 1097
                    │
                    ▼
   env 用 real_seed=1097 reset

两层好处:

  1. 可复现:同 env_version 内,env_instance 5 始终对应同一 real seed,跨 run 比较 bit-exact
  2. 抽象:agent 永远不见 real seed 值。即使 env 作者重新生成 seed 池(bump env_version),agent 代码不变 —— 它一直只用 ID

11. 沙箱与反作弊 #

沙箱实现 (src/hlbench/core/sandbox.py)

  • 子进程隔离: multiprocessing(spawn),每个 submit 一个 fresh child,持有一个 Policy 实例
  • act() 墙挂时间: 子进程内 signal.setitimer(ITIMER_REAL, act_wall_s);超时抛 _ActTimeout → 分类为 act_timeout
  • init/episode 墙挂时间: 父端 Pipe.poll(timeout);超时强制终止子进程
  • 内存上限: resource.setrlimit(RLIMIT_AS, max_rss_bytes)。Linux 上有效;macOS 下 Apple's malloc 经常忽略 → best-effort
  • Stdout/stderr 捕获 (0.1.0a1): 子进程内 sys.stdout/stderr 重定向到 StringIO,每 episode 完后 swap 回父进程,写到 ep_<XXX>/{stdout,stderr}.txt

Anti-hack (AGENTS.md §5)

  1. 禁载 pretrained 模型权重 —— 不能加载 system/ 之外、非本次 run 自己产生的模型/参数文件
  2. 禁拿 held-out seed —— 不通过任何 env 变量、文件、信道暴露;不能枚举/猜测
  3. 禁外部 compute —— policy.py 完全在沙箱内运行;禁开子进程、socket、外部服务
  4. 禁读 workspace 外 —— policy.py 只读 system/;读 feedback/、/tmp、$HOME 等都报错

禁用 import 列表 (AGENTS.md §3.2,0.1.0a1 起强制)

Sandbox 子进程在 sys.path 注入 system/ 之前,先在 sys.meta_path 装一个 _DeniedImportFinder。任何匹配 以下集合的顶层 / dotted-prefix import 触发 _DeniedImport(ImportError),被分类为 denied_import verdict。

类别模块原因
Pretrained MLtransformers, huggingface_hub, timm, diffusers载预训练权重 → 违反 Core Invariant
External LLM APIopenai, anthropic, google.genai, cohere外部模型调用 → 违反 Core Invariant
RL frameworkstable_baselines3, ray, rllib, cleanrl, sb3_contrib捆绑 checkpoints / 旁路 submit 接口
HTTP clientrequests, httpx, aiohttp触发网络访问
已知限制 AGENTS.md §3.2 还列了 urllib / socket / subprocess 等 stdlib。这些在 Python 启动期就被 multiprocessing/SSL 等被动加载到 sys.modules,粗暴驱逐会破坏 gymnasium。当前仅靠 finder 拦截「首次」import: urllib.request 等子模块(未预加载)能拦,顶层 import subprocess 拦不住。完整 stdlib 阻断需要更深的 sandbox 加固(后续工作)。

AGENTS.md §3.3 资源上限默认值

限制默认来源
act() 墙挂10 msact_wall_ms
Policy.__init__1 spolicy_load_wall_s

Env registration 可调宽(planning-heavy 任务可能需要 100 ms act);CLI 可继续覆盖。Effective values 永远在 GET /info:resource_limits。

12. 运行产物 runs/ #

所有 run artifacts 都落在 runs/<model>/<env>/<exp-id>/ 这个 三级嵌套目录下。Server 在构造时只需要 runs_root + (model, env_id, exp_id), 会自动算出 run_dir 并 mkdir。

runs/
└── <model>/                    # claude-opus-4-7 / random / ppo-1m / ...
    └── <env>/                   # pendulum / halfcheetah / breakout / ...
        └── <exp-id>/            # 用户指定 or YYYY-MM-DDTHH-MM-SS__<6-hex>
            ├── run.json         # 顶层 run metadata + final_score
            ├── workspace/       # agent 的 local view
            │   ├── AGENTS.md
            │   ├── system/
            │   └── feedback/    # 每次 submit 一个 submit_NNN/
            ├── checkpoints/     # 每个 submit 的代码快照
            │   ├── submit_000/
            │   │   ├── _meta.json       # validation_status / 文件列表 / 等
            │   │   ├── policy.py
            │   │   └── (其他 system/ 子文件)
            │   └── submit_001/
            └── logs/
                ├── harness.log  # harness 执行日志(0.1.0a1 计划)
                ├── agent.jsonl  # agent 活动日志(deferred)
                └── env.log      # env 警告(通常空)

exp-id 命名规则 (output.md §2.3)

用户可以传 exp_id="my_experiment";不传时 Server 自动生成 YYYY-MM-DDTHH-MM-SS__<6-hex>,其中 hex 部分是 SHA-256(model | env | budget | timestamp) 的前 6 位 —— 让同一秒内多个 (model, env) run 不会撞名,且确定性可复现。

run.json — 顶层元数据 (output.md §3)

读单个 run.json 必须够回答:这是哪个 run?最终分多少?所有 artifacts 在哪?所有数据指标怎样?字段组:

  • identity: model / env / exp_id / versions.{harness, env, agents_md_hash}
  • experiment_dimensions: episode_budget / min/max_per_submit / agent_harness 等
  • timing: start / end / wall_time_seconds
  • outcome.status: completed / aborted / error
  • outcome.final_score: 见 §9 打分公式
  • outcome.held_out_returns: M 个 raw return 数组(供 variance/outlier 分析)
  • outcome.auxiliary: 见 §9 表
  • artifacts: 相对路径,指向同目录的 workspace/、checkpoints/、logs/*

Logs 与 feedback 的隔离 (output.md §6)

不变量 9 logs/ 下的文件 agent 永远读不到。任何 agent 迭代时需要的诊断信息都必须经 workspace/feedback/ 投递,不能 抄近路写到 logs/。

13. 9 条设计不变量 #

这些是反复推敲、settled 的核心假设。改动其中任何一条都需要明确的用户授权 (CLAUDE.md)。

  1. Method neutrality: 不规定 heuristic vs learned。torch/jax 允许;transformers/HF/SB3 禁。
  2. Held-out 完全不可见: agent 任何时候看不见 size、seeds、单 episode return、baselines、final_score(run 进行中)。
  3. env_instance 整数寻址: agent 用 [0, n) 整数 ID;ID→real seed 映射 server-side,永不暴露。
  4. /info 是配置唯一真相: 没有 _run.json;merged static config + live state 都从这里来。
  5. Workspace 恰好 3 件: AGENTS.md / system/ / feedback/。Task 规约由 GET /task 服务,不 staging 进 workspace。
  6. system/ 是 Python 包: policy.py 是 required 入口;sys.path[0] = workspace/system/ 期间运行。
  7. Per-run server: 一个 Server 对应一个 (model, env, exp-id);沙箱不复用 run。
  8. 共享主机 + HTTP 控制面: agent 与 server 同机,共享 workspace 文件系统;agent 走 HTTP 控制,文件系统读 feedback。分布式部署永久 out of scope。
  9. Lib-first internals: 核心是 hlbench.core.Server 类;HTTP 是薄包装。Tests / dev / orchestration 可直接用 lib;agent 必须走 HTTP。

from pathlib import Path
import shutil
from hlbench.core.server import Server

srv = Server(
    env_id="pendulum",
    runs_root=Path("./runs"),
    model="reference-pd",
    exp_id="demo",          # 不传则自动 YYYY-...__
)
# 此时 ./runs/reference-pd/pendulum/demo/{workspace,checkpoints,logs}/ 已建好
# Server 暴露三个只读属性: srv.run_dir / srv.workspace_dir / srv.exp_id

shutil.copy("agents/pd_pendulum/policy.py",
            srv.workspace_dir / "system" / "policy.py")

for i in range(5):
    result = srv.submit(list(range(i*4, i*4+4)))
    print(f"submit {i}: mean_return = {result.summary['mean_return']:.1f}")

final = srv.finalize()
print(f"final_score = {final.final_score}")
print(f"run.json    = {final.run_json_path}")
# → ./runs/reference-pd/pendulum/demo/run.json

HTTP 模式 (CLI)

# 1) init: 在 runs/<model>/<env>/<exp-id>/ 建好 workspace
hlbench init --env pendulum \
  --runs-root ./runs --model reference-pd --exp-id demo

# 2) 投放策略代码 (写到 init 输出的 workspace 路径)
cp agents/pd_pendulum/policy.py \
  ./runs/reference-pd/pendulum/demo/workspace/system/policy.py

# 3) serve: 起 HTTP server (Terminal A)
hlbench serve --env pendulum \
  --run-dir ./runs/reference-pd/pendulum/demo

# 4) Drive (Terminal B) — server URL 默认 http://127.0.0.1:8765
hlbench info
hlbench submit --env-instances 0-7
hlbench submit --env-instances 8,16,32
hlbench finalize

# 5) 读分
cat ./runs/reference-pd/pendulum/demo/run.json | jq '.outcome.final_score'
